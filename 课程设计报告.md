# 课程设计报告：中国海外文物知识图谱系统

## 一、设计目的与背景

中国是文物大国，大量珍贵文物流散于海外博物馆中。目前，这些文物信息分散在各个博物馆的独立数据库中，缺乏统一的语义化关联和知识组织，导致文物研究、文化传承和公众教育面临信息孤岛问题。

本课程设计旨在构建一个**中国海外文物知识图谱系统**，以三家美国博物馆（克利夫兰艺术博物馆、尼尔森-阿特金斯艺术博物馆、宾夕法尼亚大学考古与人类学博物馆）的中国藏品为数据源，通过爬虫采集、数据清洗、机器翻译、结构化存储和知识图谱构建等环节，实现文物信息的语义化整合与关联查询，为文物研究和文化传播提供知识服务支撑。

## 二、系统总体设计

### 2.1 系统架构

系统采用**分层架构**设计，自下而上分为数据采集层、数据处理层、数据存储层和知识图谱层：

```
┌─────────────────────────────────────────────────┐
│                  知识图谱层                       │
│         Neo4j 图数据库 / Cypher 查询              │
├─────────────────────────────────────────────────┤
│                  数据存储层                       │
│         MySQL 8.0 关系数据库 / Docker 容器化       │
├─────────────────────────────────────────────────┤
│                  数据处理层                       │
│    数据清洗 → 机器翻译 → CSV 导入 → URL 代理       │
├─────────────────────────────────────────────────┤
│                  数据采集层                       │
│   Cleveland API / Nelson-Atkins 爬虫 / Penn 爬虫  │
└─────────────────────────────────────────────────┘
```

### 2.2 技术选型

| 层次 | 技术 | 说明 |
|------|------|------|
| 数据采集 | Python + Requests + Selenium + BeautifulSoup | 支持API接口和动态页面爬取 |
| 数据清洗 | Python + Pandas + 正则表达式 | 年代标准化、类型映射、去重 |
| 机器翻译 | 百度翻译API | 英文字段批量翻译为中文 |
| 关系存储 | MySQL 8.0（Docker） | 文物元数据结构化存储 |
| 图数据库 | Neo4j 5 Community（Docker） | 知识图谱三元组存储与查询 |
| 容器化 | Docker Compose | MySQL、Neo4j、Redis、MinIO 统一编排 |
| 图片代理 | Python + mysql-connector | 解决跨域访问问题，URL 重写 |

### 3.1 数据采集模块

#### 3.1.1 克利夫兰艺术博物馆爬虫

克利夫兰博物馆提供 Open Access API（`https://openaccess-api.clevelandart.org`），支持分页查询和 JSON 格式返回。

**关键技术：**

- **分页爬取**：通过 `limit` 和 `skip` 参数实现分页，每页 100 条
- **文化过滤**：仅采集 `culture` 字段包含 "China" 或 "Chinese" 的中国文物
- **UA 轮换**：维护 User-Agent 池，随机选择请求头，降低被封禁风险
- **断点续爬**：将 `skip` 和 `count` 保存到 `progress.json`，支持中断后恢复
- **多线程图片下载**：使用 `ThreadPoolExecutor`（10 线程）并发下载文物图片

```python
url = f"https://openaccess-api.clevelandart.org/api/artworks/?limit={limit}&skip={skip}"
culture = str(item.get("culture", ""))
if "China" not in culture and "Chinese" not in culture:
    continue
```

#### 3.1.2 尼尔森-阿特金斯艺术博物馆爬虫

该博物馆无公开 API，需通过 HTML 页面解析获取数据，且网站使用 Cloudflare 防护。

**关键技术：**

- **详情页深度解析**：从列表页获取文物链接，逐个访问详情页提取完整信息
- **结构化数据提取**：结合 `ld+json` 脚本和 `.detailField` 标签两种策略提取字段
- **图片多策略获取**：优先查找高清原图下载链接，回退到 Open Graph 元数据图片

### 3.2 数据清洗模块

原始爬取数据存在大量格式不统一、中英混杂、缺失值等问题，需要进行系统化清洗。

#### 3.2.1 年代标准化

这是数据清洗中最复杂的部分。原始 `period` 字段包含多种英文表述格式：

| 原始格式 | 标准化结果 |
|---------|-----------|
| 14th century | 14世纪 |
| late 15th century | 15世纪末 |
| first half 14th century | 14世纪上半叶 |
| Qing Dynasty (1644-1911) | 1644—1911清 |
| Kangxi period | 康熙年间 |
| ca. 1700 | 约1700年 |
| 1368-1644 | 1368—1644 |
| 2nd century BCE | 公元前2世纪 |

**实现策略：**

1. **BCE/CE 转换**：将 `BCE`、`BC` 转为"公元前"，`CE`、`AD` 转为"公元"
2. **世纪转换**：将 `14th century` 转为"14世纪"，支持 `early/late/mid` 前缀
3. **朝代映射**：建立英文朝代名到中文的映射表（Qing→清、Ming→明等 20+ 条）
4. **帝王年号映射**：Kangxi→康熙、Qianlong→乾隆等 17 个年号
5. **噪音词清理**：移除 "period"、"of the"、"painting" 等无意义词汇
6. **简写年份补齐**：`1368—68` 自动补齐为 `1368—1368`

```python
dynasty_map = {
    'Northern Song': '北宋', 'Southern Song': '南宋',
    'Qing': '清', 'Ming': '明', 'Yuan': '元', ...
}
era_map = {
    'Kangxi': '康熙', 'Qianlong': '乾隆', 'Yongzheng': '雍正', ...
}
```

#### 3.2.2 类型中文映射

将英文文物类型映射为标准中文分类：

```python
type_mapping = {
    'Ceramic': '陶瓷', 'Jade': '玉器', 'Painting': '绘画',
    'Sculpture': '雕塑', 'Textile': '纺织品', 'Print': '版画/印刷品',
    'Embroidery': '刺绣', 'Lacquer': '漆器', ...
}
```

映射结果保存为 `type_cn` 列，在后续导入时优先使用。

#### 3.2.3 智能去重

采用两级去重策略：

1. **精确去重**：基于 `accession_number`（馆藏编号）去重，保留首次出现的记录
2. **模糊去重**：对无馆藏编号的记录，将标题标准化（去标点、转小写）后去重

#### 3.2.4 图片链接验证

使用多线程（10 线程）对图片 URL 发送 HEAD 请求验证有效性，过滤无效链接。

### 3.3 机器翻译模块

使用百度翻译 API 将英文字段批量翻译为中文。

**关键技术：**

- **批量翻译**：将多条文本用换行符拼接后一次性请求，减少 API 调用次数
- **断点续传**：定期保存翻译进度，中断后可从上次位置继续
- **MD5 签名认证**：按照百度 API 规范计算 `appid + q + salt + secret_key` 的 MD5 签名
- **容错处理**：翻译失败时返回原文，避免程序崩溃

```python
q = "\n".join(sentences)
sign_str = appid + q + salt + secret_key
sign = hashlib.md5(sign_str.encode()).hexdigest()
```

### 3.5 知识图谱构建模块

#### 3.5.1 图谱模式设计

知识图谱采用以下节点和关系设计：

**节点类型（8 种）：**

| 节点标签 | 主键 | 属性 | 说明 |
|---------|------|------|------|
| Artifact | object_id | title, dimensions, description, credit_line, accession_number | 文物 |
| ObjectID | value | - | 文物编号 |
| Period | name | - | 朝代/时期 |
| Material | name | - | 材质 |
| ArtifactType | name | - | 文物品类 |
| Museum | name | name_en | 博物馆 |
| Location | name | - | 地理位置 |

**关系类型（7 种）：**

| 关系 | 起点 → 终点 | 说明 |
|------|-------------|------|
| 编号 | Artifact → ObjectID | 文物的唯一编号 |
| 所属朝代 | Artifact → Period | 文物所属朝代 |
| 制作材质 | Artifact → Material | 文物制作材质 |
| 文物品类 | Artifact → ArtifactType | 文物类别 |
| 收藏馆藏 | Artifact → Museum | 收藏该文物的博物馆 |
| 坐落地址 | Museum → Location | 博物馆所在地址 |

**设计决策说明：**

1. **Artifact 以 object_id 为主键**：确保节点唯一性，title 作为属性存储
2. **ObjectID 独立节点**：将编号从属性提升为节点，支持按编号的图查询
3. **dimensions/description 作为属性**：这些是文物的独有属性，不会被其他节点共享，适合作为属性而非独立节点
4. **材质/品类使用 MERGE**：同名材质/品类只创建一个节点，多文物共享，体现知识图谱的关联优势

#### 3.5.2 三元组构建程序（createTriple.py）

**核心流程：**

1. 从 MySQL 读取文物数据（含博物馆关联信息）
2. 清空 Neo4j 旧数据（`MATCH (n) DETACH DELETE n`）
3. 创建索引（7 个索引，加速查询）
4. 逐条构建三元组

**关键实现：**

- **MERGE 语义**：使用 `MERGE` 而非 `CREATE`，确保同一实体只创建一次
- **多值字段拆分**：材质按 `|` 和 `;` 分割，品类按 `|`、`;`、`、` 分割
- **朝代多行处理**：`period` 字段可能包含换行分隔的多个朝代，逐行创建关系
- **ON CREATE / ON MATCH**：确保属性在创建和匹配时都能正确更新

```python
session.run("""
    MERGE (a:Artifact {object_id: $artifact_id})
    ON CREATE SET a.title = $title, a.dimensions = $dimensions, ...
    ON MATCH SET  a.title = $title, a.dimensions = $dimensions, ...
""", {...})

session.run("""
    MERGE (oid:ObjectID {value: $artifact_id})
    WITH oid
    MATCH (a:Artifact {object_id: $artifact_id})
    MERGE (a)-[:编号]->(oid)
""", {'artifact_id': artifact_id})
```

### 3.6 图片 URL 代理模块（changeURL.py）

#### 3.6.1 问题背景

文物图片原始 URL 指向各博物馆的 CDN 服务器（如 `openaccess-cdn.clevelandart.org`），前端直接访问时存在**跨域问题（CORS）**：浏览器因同源策略限制，无法从本系统域名加载第三方图片资源。

#### 3.6.2 解决方案

将图片下载到云服务器本地，并将数据库中的 `image_url` 和 `image_path` 重写为服务器本地路径：

| 字段 | 修改前 | 修改后 |
|------|--------|--------|
| image_url | `https://openaccess-cdn.clevelandart.org/...` | `http://39.106.231.119/images/{object_id}.jpg` |
| image_path | `images/cleveland\149146.jpg` | `/var/www/image/{object_id}.jpg` |

#### 3.6.3 实现细节

```python
SERVER_IP = "39.106.231.119"
IMAGE_URL_PREFIX = f"http://{SERVER_IP}/images/"

update_sql = """
    UPDATE artifact 
    SET image_url = %s, image_path = %s 
    WHERE object_id = %s
"""
cursor.execute(update_sql, (image_url, f"/var/www/image/{object_id}.jpg", object_id))
```

通过将图片托管在自有服务器上，前端请求同域资源，彻底解决跨域问题。

### 3.7 容器化部署

#### 3.7.1 Docker Compose 编排

系统使用 Docker Compose 统一管理四个服务：

```yaml
services:
  mysql:
    image: mysql:8.0
    command: --default-authentication-plugin=mysql_native_password
    ports: ["3306:3306"]
    volumes:
      - dev_mysql_data:/var/lib/mysql
      - ./mysql/init.sql:/docker-entrypoint-initdb.d/init.sql

  neo4j:
    image: neo4j:5-community
    ports: ["7474:7474", "7687:7687"]

  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]

  minio:
    image: minio/minio:latest
    command: server /data --console-address ":9001"
    ports: ["9000:9000", "9001:9001"]
```

**关键配置：**

- MySQL 添加 `--default-authentication-plugin=mysql_native_password`，兼容 pymysql 认证
- MySQL 挂载 `init.sql` 到初始化目录，容器首次启动时自动建表
- Neo4j 通过 `NEO4J_AUTH` 环境变量设置认证信息
- 所有服务通过 `dev-net` 桥接网络互通

#### 3.7.2 本地开发与云端部署

- **本地开发**：使用 `docker-compose.dev.yml` 在本地启动所有服务，进行数据采集、清洗和图谱构建
- **云端部署**：将构建好的数据和应用上传至云服务器（阿里云 ECS，IP: 39.106.231.119），通过 Nginx 反向代理对外提供服务
